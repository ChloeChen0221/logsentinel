# LogSentinel 架构升级方案（毕设裁剪版）

> 目标：从单机 Demo 演进为**可水平扩展的生产级日志告警平台**，聚焦「水平扩展」主线，砍掉偏离主题或过度工程的部分。

---

## 零、部署环境与约束

### 部署环境

| 项 | 配置 |
|---|---|
| **平台** | 腾讯云 Linux 开发机（单节点） |
| **OS** | TencentOS Server 3.2（CentOS 系，内核 5.4） |
| **K8s 发行版** | K3s v1.34.3（单节点集群） |
| **容器运行时** | containerd 2.1.5 |
| **部署方式** | Helm Umbrella Chart |
| **第三方组件获取** | `helm pull` 下载到本地 `helm-charts/` 目录，定制 `values.yaml` |
| **入口** | Kubernetes Service（NodePort / LoadBalancer），不使用 Nginx/Ingress |

### 单节点约束下的设计调整

1. **应用层水平扩展**：Worker × 3，API × 2（Pod 级水平扩展，单节点多 Pod）
2. **调度亲和性**：`podAntiAffinity` 使用 `preferredDuringSchedulingIgnoredDuringExecution`（软约束），保证单节点下 Pod 仍可调度
3. **资源隔离**：每个 Pod 配置 `resources.requests/limits`，避免 Pod 间 CPU/内存竞争影响压测结论
4. **第三方组件单副本**：PostgreSQL / Redis / Loki 均为单副本部署，HA 方案列入未来工作
5. **论文定位**：**架构设计层面支持多节点水平扩展，单节点部署作为验证环境**；答辩时强调分片、分布式锁、状态外置等机制在多节点场景下可无缝扩展

## 一、裁剪说明

### 被砍掉的内容及理由

| 原方案 | 砍掉理由 | 替代方案 |
|---|---|---|
| **pg_partman 月份分区 + MinIO 冷归档** | 压测数据量几百万行，PG 单表扛得住，分区无收益；环境复杂度翻倍；偏离水平扩展主线 | 论文"未来工作"一句话带过 |
| **Celery + Redis/RabbitMQ Broker + 死信队列** | 另一技术栈，与纯 asyncio 架构冲突；容器数量 +3；`asyncio.Queue` 即可实现解耦+重试 | `asyncio.Queue` + 后台 task |
| **Silence + Inhibit + 分组聚合** | Alertmanager 已成熟，讲不出创新点；偏离水平扩展主线；现有冷却机制覆盖 80% 场景 | 答辩话术："告警治理作为后续工作" |
| **Redis Cluster（3 主 3 从）** | Cluster 模式下分布式锁、pub/sub 有跨 slot 坑；单节点够用 | 单节点 Redis（或 Sentinel 1 主 2 从） |
| **PostgreSQL 主从读写分离** | 毕设读 QPS 不需要分流；SQLAlchemy 读写分离易出一致性问题 | 单实例 PG + 连接池 + 索引 |
| **Loki 微服务模式部署** | Loki 微服务模式给 TB 级日志用；毕设 single-binary 够用；讲 Loki 偏题 | Loki single-binary + filesystem |
| **K8s HPA 自动扩缩容** | 压测场景规则数量固定，不需要动态伸缩；HPA 指标调优复杂，答辩容易被追问；副本数变化会触发频繁 rebalance 影响压测数据稳定性 | 固定副本数（Worker × 3，API × 2） |

### 保留的核心改造

**主线：水平扩展 + 状态外置 + 路径异步**

三阶段渐进式升级：
- **P1**：存储层升级（SQLite → PG + Redis）
- **P2**：Worker 水平扩展（分片 + 分布式锁 + 并发评估）
- **P3**：路径异步化（Loki aiohttp + 进程内通知队列 + WebSocket 推送）

## 四、P2：Worker 水平扩展

### 4.1 分片策略：一致性哈希 + 心跳

**设计思路：**
1. 每个 Worker 启动时向 Redis 注册自己（5s 续期心跳，15s 无心跳视为死亡）
2. 每轮评估前，查询所有存活 Worker 列表
3. 对每条规则计算 `hash(rule_id) % len(workers)`，只处理属于自己的规则
4. Worker 增减时自动 rebalance（15s 内收敛）

**关键代码**（`backend/engine/sharding.py` 新增）：

```python
class RuleSharding:
    """基于 Redis 的规则分片与 Worker 心跳"""

    def __init__(self, redis, worker_id: str, heartbeat_ttl: int = 15):
        self.redis = redis
        self.worker_id = worker_id
        self.ttl = heartbeat_ttl

    async def heartbeat(self):
        """每轮评估开始前调用，续期心跳"""
        await self.redis.set(f"worker:{self.worker_id}", "1", ex=self.ttl)

    async def alive_workers(self) -> list[str]:
        """获取当前存活的 Worker 列表（按 ID 排序保证一致性）"""
        keys = await self.redis.keys("worker:*")
        return sorted(k.decode().split(":", 1)[1] for k in keys)

    def owns(self, rule_id: int, workers: list[str]) -> bool:
        """一致性哈希决定规则归属"""
        if not workers:
            return False
        return workers[rule_id % len(workers)] == self.worker_id
```

### 4.2 Worker 主循环改造

```python
# backend/engine/worker.py
async def _execute_cycle(self):
    await self.sharding.heartbeat()
    workers = await self.sharding.alive_workers()
    rules = await evaluator.load_enabled_rules()

    # 1. 分片过滤
    my_rules = [r for r in rules if self.sharding.owns(r.id, workers)]

    # 2. 并发评估（替代原串行 for 循环）
    sem = asyncio.Semaphore(self.concurrency)  # 默认 10

    async def _eval_with_lock(rule):
        async with sem:
            # 分布式锁兜底：防 rebalance 瞬间重复执行
            lock_key = f"lock:rule:{rule.id}"
            got = await self.redis.set(
                lock_key, self.worker_id, nx=True, ex=60
            )
            if not got:
                return  # 其他 Worker 正在处理
            try:
                await evaluator.evaluate_rule(rule)
            finally:
                await self.redis.delete(lock_key)

    await asyncio.gather(
        *[_eval_with_lock(r) for r in my_rules],
        return_exceptions=True,
    )
```

### 4.3 窗口计数器外置到 Redis

**问题：** 当前 `evaluator.window_counters` 是内存 dict，Worker 重启丢失，多副本无法共享。

**改造：**

```python
# backend/engine/window_counter.py（新增）
class RedisWindowCounter:
    """基于 Redis HASH 的滑动窗口计数器"""

    async def incr(self, rule_id: int, fingerprint: str,
                   count: int, window_seconds: int) -> int:
        key = f"window:{rule_id}:{fingerprint}"
        pipe = self.redis.pipeline()
        pipe.hincrby(key, "count", count)
        pipe.expire(key, window_seconds + 10)  # 窗口长度+10s 缓冲
        results = await pipe.execute()
        return results[0]

    async def get(self, rule_id: int, fingerprint: str) -> int:
        key = f"window:{rule_id}:{fingerprint}"
        val = await self.redis.hget(key, "count")
        return int(val) if val else 0
```

## 五、P3：路径异步化

### 5.1 Loki 查询：同步 → 异步 + 缓存 + 限流

**问题：** 当前 `loki_client.py` 用 `requests.get()` 在异步函数里，阻塞事件循环。

**改造：** 改用 `aiohttp` + `aiolimiter` + Redis 缓存

```python
# backend/engine/loki_client.py
import aiohttp
import hashlib
import json
from aiolimiter import AsyncLimiter

class LokiClient:
    def __init__(self, base_url, redis, max_qps: int = 50, timeout: int = 10):
        self.base_url = base_url
        self.redis = redis
        self.limiter = AsyncLimiter(max_qps, 1)  # 50 QPS 上限
        self.timeout = aiohttp.ClientTimeout(total=timeout)
        self._session: aiohttp.ClientSession | None = None

    async def start(self):
        self._session = aiohttp.ClientSession(timeout=self.timeout)

    async def query_range(self, query: str, start: int, end: int):
        # 1. 缓存命中（相同查询 10s 内复用）
        cache_key = f"loki:{hashlib.md5(f'{query}{start}{end}'.encode()).hexdigest()}"
        cached = await self.redis.get(cache_key)
        if cached:
            return json.loads(cached)

        # 2. 限流
        async with self.limiter:
            # 3. 异步 HTTP + 重试退避
            for attempt in range(3):
                try:
                    async with self._session.get(
                        f"{self.base_url}/loki/api/v1/query_range",
                        params={"query": query, "start": start, "end": end},
                    ) as resp:
                        resp.raise_for_status()
                        data = await resp.json()
                        await self.redis.setex(cache_key, 10, json.dumps(data))
                        return data
                except (asyncio.TimeoutError, aiohttp.ClientError) as e:
                    if attempt == 2:
                        raise
                    await asyncio.sleep(2 ** attempt)  # 指数退避
```

### 5.2 通知路径：进程内 asyncio.Queue（不用 Celery）

**问题：** 当前通知在 `evaluator` 中同步调用，失败会阻塞规则评估。

**方案：** 生产者-消费者模式，用 `asyncio.Queue` 解耦

```python
# backend/notifier/queue.py（新增）
import asyncio
import structlog

logger = structlog.get_logger()

# 全局通知队列（每个 Worker 进程一个）
notification_queue: asyncio.Queue = asyncio.Queue(maxsize=10000)


async def enqueue_notification(alert_id: int, rule_name: str, channel: str):
    """生产者：evaluator 调用此方法，立即返回"""
    try:
        notification_queue.put_nowait({
            "alert_id": alert_id,
            "rule_name": rule_name,
            "channel": channel,
        })
    except asyncio.QueueFull:
        logger.error("notification_queue_full", alert_id=alert_id)


async def notifier_worker():
    """消费者：独立协程，负责发送通知 + 重试"""
    while True:
        task = await notification_queue.get()
        try:
            for attempt in range(3):
                try:
                    await _send(task)
                    await _record_success(task)
                    break
                except Exception as e:
                    if attempt == 2:
                        logger.error("notify_failed_final", task=task, error=str(e))
                        await _record_failure(task, str(e))
                    else:
                        await asyncio.sleep(2 ** attempt)  # 1s, 2s, 4s
        finally:
            notification_queue.task_done()
```

**Worker 启动时：**

```python

    # 启动 N 个通知消费者协程
```

| 阶段 | 内容 | 工作量评估 | 前置 |
|---|---|---|---|
| **P1: 存储升级** | SQLite → PG + Redis，索引/连接池/alembic 迁移 | ~3 天 | - |

## 七、验证与压测

### 压测场景设计

| 场景 | 指标 | 对比 |
|---|---|---|
| **规则评估吞吐** | 单位时间评估规则数 | Worker 1 副本 vs 3 副本 |
| **告警查询延迟** | P99 响应时间 | SQLite vs PG + 索引 |
| **Loki 查询 QPS** | 实际发出的 QPS | 无缓存 vs 10s 缓存命中率 |
| **通知吞吐** | 每秒处理通知数 | 同步调用 vs asyncio.Queue |
| **Worker 故障恢复** | 重平衡耗时 | 杀掉 1 个副本，观察规则重分配时间 |

### 论文可讲的亮点

1. **水平扩展**：一致性哈希 + 分布式锁实现 Worker 无状态水平扩展
2. **状态外置**：将进程内状态（窗口计数器、冷却状态）迁移到 Redis，消除单点
3. **异步解耦**：生产者-消费者模式解耦规则评估与通知发送
4. **查询优化**：Loki 查询缓存 + 限流降低后端压力
5. **实时性**：WebSocket 推送告警，替代前端轮询
