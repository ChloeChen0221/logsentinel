# 目标能力与系统设计方案

本平台旨在实现一个轻量但具备工程亮点的 **日志告警规则引擎系统**，基于 Loki 日志系统构建规则检测、告警生成与通知闭环。

---

## 1. 目标能力

### 1.1 规则能力（3 类核心规则）

系统支持三类规则，覆盖最常见的日志告警场景：

- **Pattern 规则**
  - 关键词匹配 / 正则匹配 / 日志级别过滤
  - 用于简单错误检测

- **Rate 规则**
  - 在时间窗口内出现次数 ≥ 阈值触发告警（最常见）

- **Sequence 规则**
  - 简单事件关联检测  
  - 规则形式：A 出现后 T 秒内必须出现 B，否则报警

---

### 1.2 告警能力

平台具备完整告警工程能力：

- **去重（Dedup）**
  - 同一问题合并为一条告警，避免刷屏

- **冷却（Cooldown）**
  - 触发后 N 分钟内只更新计数，不重复通知

- **静默（Silence）**
  - 按规则/标签在指定时间段内不通知

- **确认与恢复（Ack / Resolve）**
  - 告警状态流转，前端可手动操作

---

## 2. 总体架构设计（三服务拆分）

系统架构建议拆为三个核心服务，职责清晰，便于论文描述与工程实现：

---

### 2.1 API 服务（FastAPI）

负责平台管理与对外接口：

- 规则 CRUD 管理
- 告警查询与状态管理
- 静默策略管理
- 通知渠道配置
- 简单鉴权（Token 即可）

---

### 2.2 Engine 服务（规则引擎 Worker）

核心规则执行组件：

- 周期性从 Loki 拉取日志
- 执行规则匹配、窗口计数、序列检测
- 去重 / 冷却 / 静默控制
- 产出告警记录与通知任务

---

### 2.3 Notifier 服务（可选独立）

通知模块，可独立部署也可并入 Engine：

- Email 通知
- 钉钉 Webhook
- 企业微信等

附加工程能力：

- 失败重试
- 通知限流

## 3. 规则表达设计（JSON 表达，前端天然适配）

规则采用 JSON 结构定义，便于表单编辑与存储。

### 3.1 示例：Rate 规则结构

```json
{
  "name": "OOM频发",
  "enabled": true,
  "type": "rate",
  "selector": {
    "namespace": "demo",
    "labels": {"app": "myservice"}
  },
  "prefilter": {
    "logql": "{namespace=\"demo\", app=\"myservice\"} |= \"OOMKilled\""
  },
  "match": {
    "mode": "regex",
    "pattern": "OOMKilled|OutOfMemory"
  },
  "window_seconds": 60,
  "threshold": 3,
  "group_by": ["namespace","pod"],
  "severity": "P1",
  "cooldown_seconds": 300,
  "channels": ["email","dingtalk"]
}
````

---

### 3.2 关键设计点说明

* **prefilter.logql**

  * Loki 先筛选候选日志，减轻引擎压力

* **match**

  * Python 中做最终匹配（正则/字段判断）

* **group_by**

  * 决定去重维度，如按 pod 或 namespace 聚合

* **cooldown**

  * 工程告警必备功能，防止刷屏

---

### 3.3 Sequence 规则扩展结构

Sequence 规则保持同风格：

* `type: "sequence"`
* `steps: [A, B]`
* `max_gap_seconds`

---

## 4. Loki 日志拉取机制（游标 Cursor）

避免每次查询最近窗口导致重复处理，推荐使用游标机制。

### 4.1 推荐方式

每条规则维护一个 cursor（上次查询到的时间戳）：

引擎循环流程：

1. 读取规则 cursor_ts
2. Loki query_range(cursor_ts → now) 拉取日志
3. 处理规则匹配
4. 更新 cursor 写回 DB

---

### 4.2 优势

* 逻辑单调，不漏不重
* 重复处理极少
* 易于性能实验：吞吐、延迟可测

---

## 5. 状态管理（规则引擎核心）

系统需维护运行状态，建议存 Redis（开发期可先内存）。

---

### 5.1 Rate 规则：滑动窗口计数

对每个 fingerprint（rule_id + group_by）维护：

* 时间戳队列（deque）
* 新命中 append
* 窗口外 pop
* 队列长度 ≥ threshold 触发告警

---

### 5.2 Sequence 规则：简单状态机

对每个 fingerprint 保存：

* 当前 step index
* 上一步命中时间
* 超过 max_gap 秒自动 reset

---

## 6. 告警数据模型设计

两张主表结构清晰，论文与前端都好实现。

---

### 6.1 alerts（告警主表）

记录去重后的告警实例：

* id
* rule_id
* fingerprint
* severity
* status（open/ack/resolved）
* first_seen, last_seen
* hit_count
* title, summary
* labels（json）

---

### 6.2 alert_events（命中明细，可选加分）

记录每次规则命中的细节：

* id
* alert_id
* ts
* sample_log（样例日志）
* matched_fields（json）

前端告警详情页可展开查看命中记录。

---

## 7. 通知模块设计（插件式）

统一接口：

