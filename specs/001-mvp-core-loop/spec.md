# Feature Specification: MVP 核心闭环 - K8s 日志告警平台

**Feature Branch**: `001-mvp-core-loop`  
**Created**: 2026-02-04  
**Status**: Draft  
**Input**: 请为本项目生成一个"第一阶段 MVP"规格（spec），目标是最小可用闭环：从 K8s 产生测试日志 -> 被查询到 -> 被规则匹配 -> 生成告警 -> 在 Web 界面可见，并完成一次基础通知。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建关键词匹配规则并触发告警 (Priority: P1)

作为平台管理员，我需要创建一条基于关键词的告警规则（例如日志包含 "ERROR"），当 Kubernetes 集群中的 Pod 输出匹配的日志时，系统能够自动生成告警并在 Web 界面展示，这样我就能及时发现集群中的错误日志。该规则以平台定义的统一规则模型进行描述，规则逻辑由系统内置的规则引擎解析与执行，而非写死在代码中。

**Why this priority**: 这是整个告警平台的核心价值链，验证了从日志采集、规则匹配、告警生成到界面展示的完整闭环，是 MVP 的基础。

**Independent Test**: 可以通过在 demo 命名空间部署一个持续输出 "ERROR" 日志的测试 Pod，创建关键词规则后，在 Web 界面查看是否生成告警记录来独立验证。

**Acceptance Scenarios**:

1. **Given** 系统已连接到 Loki 并能查询到 demo 命名空间的日志，**When** 管理员创建一条规则（名称="测试错误告警"，关键词="ERROR"，命名空间="demo"，严重级别="high"），**Then** 规则保存成功并显示在规则列表中，状态为"已启用"
2. **Given** 已创建关键词规则且已启用，**When** demo 命名空间中的 Pod 输出包含 "ERROR" 的日志，**Then** 系统在 60 秒内生成一条新告警，告警列表中显示该告警（包含规则名称、严重级别、命中次数=1、首次触发时间）
3. **Given** 已生成的告警记录，**When** 管理员点击告警查看详情，**Then** 显示告警详细信息（规则名称、严重级别、首次触发时间、最后触发时间、累计命中次数、至少一条匹配的日志样例）

---

### User Story 2 - 创建窗口阈值规则并观察告警合并 (Priority: P2)

作为平台管理员，我需要创建一条基于时间窗口和阈值的规则（例如 60 秒内命中 3 次），当规则触发后，同一来源的多次命中能够合并到同一条告警记录中（更新命中次数和最后触发时间），避免告警泛滥。

**Why this priority**: 窗口阈值规则是实际生产环境中常用的告警策略，告警合并机制能有效减少告警噪音，是 MVP 必需的核心功能。

**Independent Test**: 可以通过部署一个快速输出错误日志的测试 Pod（每 10 秒输出一次 ERROR），创建窗口阈值规则（60 秒/3 次），观察告警记录的命中次数和时间更新来独立验证。

**Acceptance Scenarios**:

1. **Given** 系统已连接到 Loki，**When** 管理员创建窗口阈值规则（名称="高频错误告警"，关键词="ERROR"，命名空间="demo"，窗口=60 秒，阈值=3 次，分组维度="namespace+pod"），**Then** 规则保存成功并启用
2. **Given** 已创建窗口阈值规则，**When** demo 命名空间中同一 Pod 在 60 秒内输出 3 条包含 "ERROR" 的日志，**Then** 系统生成一条告警，命中次数=3
3. **Given** 已生成的告警（基于 namespace+pod 分组），**When** 同一 Pod 再次输出匹配日志，**Then** 告警记录更新（命中次数+1，最后触发时间更新为最新时间），而不是创建新的告警记录
4. **Given** 已生成的告警，**When** 不同 Pod（但同一命名空间）输出匹配日志，**Then** 系统创建新的独立告警记录（因为分组维度不同）

---

### User Story 3 - 规则管理与启用/停用 (Priority: P2)

作为平台管理员，我需要能够查看所有规则列表、编辑规则配置、启用或停用规则，以便灵活管理告警策略。

**Why this priority**: 规则的生命周期管理是配置化告警系统的基础能力，必须在 MVP 阶段实现，确保用户能够动态调整告警策略。

**Independent Test**: 可以通过创建多条规则、修改规则参数、切换启用状态，并验证规则引擎是否按照最新配置执行来独立验证。

**Acceptance Scenarios**:

1. **Given** 系统中已有多条规则，**When** 管理员访问规则列表页面，**Then** 显示所有规则（包含规则名称、类型、严重级别、启用状态、创建时间）
2. **Given** 规则列表中的某条规则，**When** 管理员点击编辑并修改规则参数（如关键词、阈值），**Then** 修改保存成功，规则引擎使用新配置进行匹配
3. **Given** 已启用的规则，**When** 管理员停用该规则，**Then** 规则状态变为"已停用"，规则引擎不再对该规则进行匹配和告警生成
4. **Given** 已停用的规则，**When** 管理员重新启用该规则，**Then** 规则状态变为"已启用"，规则引擎恢复对该规则的匹配
5. **Given** 规则列表中的某条规则，**When** 管理员删除该规则，**Then** 规则从列表中移除，相关的历史告警记录保留但标注规则已删除

---

### User Story 4 - 基础通知机制与冷却期 (Priority: P3)

作为平台管理员，我需要在告警首次触发时收到通知，并且在冷却期内不重复收到相同告警的通知，冷却期结束后如果告警再次触发则再次通知，这样既能及时响应问题，又能避免通知轰炸。

**Why this priority**: 通知是告警闭环的最后一环，冷却期机制能有效控制通知频率。MVP 阶段允许使用简单的控制台输出作为通知方式，后续再扩展到邮件、钉钉等渠道。

**Independent Test**: 可以通过创建带冷却期的规则（如 300 秒），触发告警后观察控制台输出，等待冷却期结束后再次触发，验证通知行为来独立验证。

**Acceptance Scenarios**:

1. **Given** 已创建规则（冷却期=300 秒），**When** 规则首次触发生成告警，**Then** 系统执行一次通知动作（控制台输出告警信息：规则名称、严重级别、触发时间、匹配日志样例）
2. **Given** 告警已触发并通知，处于冷却期内，**When** 同一告警再次命中（命中次数增加），**Then** 系统更新告警记录但不执行通知动作
3. **Given** 告警的冷却期已结束，**When** 该告警再次命中，**Then** 系统执行新的通知动作，并重置冷却期计时
4. **Given** 系统运行中，**When** 查看通知日志，**Then** 能够看到所有已发送通知的记录（告警 ID、通知时间、通知内容）

---

### Edge Cases

- **规则配置错误**：当管理员创建规则时输入无效的正则表达式或负数阈值，系统应在保存前进行验证并提示错误信息
- **Loki 连接失败**：当规则引擎查询 Loki 时连接超时或失败，系统应记录错误日志，跳过本次查询周期，并在下一周期重试
- **日志查询无结果**：当规则匹配的命名空间或标签选择器没有对应的日志数据时，系统不生成告警，不报错
- **告警记录过多**：当告警列表中累积大量历史告警时，Web 界面应支持分页展示，默认按最后触发时间倒序排列
- **规则删除后的告警**：当规则被删除后，已生成的告警记录仍然保留，但在告警详情中标注"规则已删除"
- **并发规则匹配**：当多条规则同时匹配同一条日志时，系统应为每条规则分别生成或更新告警记录
- **时区处理**：所有时间戳（首次触发、最后触发）应使用统一的时区（默认 UTC），在 Web 界面展示时转换为本地时区

## Requirements *(mandatory)*

### Functional Requirements

**规则管理**

- **FR-001**: 系统必须支持创建告警规则，规则包含以下字段：规则名称（name）、启用状态（enabled）、严重级别（severity：low/medium/high/critical）、日志选择器（selector）、匹配条件（match）、窗口时间（window_seconds）、阈值（threshold）、分组维度（group_by）、冷却时间（cooldown_seconds）
- **FR-002**: 日志选择器（selector）必须至少支持按命名空间（namespace）过滤，可选支持按标签（label）和应用名（app）过滤
- **FR-003**: 匹配条件（match）必须支持关键词包含匹配（contains），可选支持正则表达式匹配（regex）
- **FR-004**: 系统必须支持查看所有规则列表，展示规则名称、类型、严重级别、启用状态、创建时间
- **FR-005**: 系统必须支持编辑已有规则的所有配置字段，修改后立即生效
- **FR-006**: 系统必须支持启用或停用规则，停用的规则不参与日志匹配和告警生成
- **FR-007**: 系统必须支持删除规则，删除后不再生成新告警，但历史告警记录保留
- **FR-008**: 系统必须对规则配置进行验证：规则名称不能为空，窗口时间和阈值必须为正整数，正则表达式必须合法

**告警生成与管理**

- **FR-009**: 规则引擎必须定期（默认每 30 秒）查询 Loki 获取符合规则选择器的日志数据
- **FR-010**: 对于关键词匹配规则，当日志内容包含指定关键词时，系统必须生成或更新告警
- **FR-011**: 对于窗口阈值规则，当指定时间窗口内匹配日志数量达到或超过阈值时，系统必须生成或更新告警
- **FR-012**: 系统必须根据规则的分组维度（group_by）对告警进行去重合并，相同分组维度的多次命中更新同一条告警记录（增加命中次数、更新最后触发时间）
- **FR-013**: 分组维度（group_by）必须至少支持按"namespace+pod"组合分组
- **FR-014**: 告警记录必须包含以下字段：告警 ID、规则 ID、规则名称、严重级别、状态（active，MVP 阶段所有告警保持 active）、首次触发时间（first_seen）、最后触发时间（last_seen）、累计命中次数（hit_count）、分组维度值、最新一条匹配日志的关键字段（时间戳、内容、Pod 名称、命名空间）
- **FR-015**: 系统必须支持查看告警列表，展示规则名称、严重级别、状态、命中次数、最后触发时间，默认按最后触发时间倒序排列
- **FR-016**: 系统必须支持分页展示告警列表，每页显示数量可配置（默认 20 条）
- **FR-017**: 系统必须支持查看告警详情，展示告警的所有字段信息和最新一条匹配的日志样例（包含时间戳、日志内容、来源 Pod、命名空间）

**基础通知**

- **FR-018**: 当告警首次触发时，系统必须执行一次通知动作
- **FR-019**: 通知内容必须包含：规则名称、严重级别、触发时间、至少一条匹配的日志样例
- **FR-020**: MVP 阶段通知方式为控制台输出（标准输出或日志文件），输出格式为结构化日志（JSON 或带时间戳的文本）
- **FR-021**: 系统必须实现冷却期机制，在冷却期内同一告警的后续命中不触发通知，仅更新告警记录
- **FR-022**: 冷却期结束后，如果告警再次命中，系统必须执行新的通知动作并重置冷却期计时
- **FR-023**: 系统必须记录所有通知历史，包含告警 ID、通知时间、通知内容

**系统集成**

- **FR-024**: 系统必须能够连接到 Loki 实例，支持配置 Loki 的 URL 地址
- **FR-025**: 系统必须使用 LogQL 查询语言从 Loki 获取日志数据
- **FR-026**: 当 Loki 连接失败或查询超时时，系统必须记录错误日志并在下一周期重试，不中断规则引擎运行
- **FR-027**: 系统必须提供 Web 界面，支持规则管理和告警查看功能
- **FR-028**: Web 界面必须支持在本地 Mac 环境的浏览器中访问

### Key Entities

- **Rule（告警规则）**: 定义日志匹配条件和告警触发逻辑的配置实体，包含规则名称、启用状态、严重级别、日志选择器（命名空间、标签）、匹配条件（关键词或正则）、窗口时间、阈值、分组维度、冷却时间等属性，Rule 的语义与执行逻辑由平台内置规则引擎统一定义，用户仅通过配置参与规则实例化。
- **Alert（告警记录）**: 规则触发后生成的告警实例，包含告警 ID、关联的规则 ID、严重级别、状态、首次触发时间、最后触发时间、累计命中次数、分组维度值、匹配的日志样例等属性，与 Rule 是多对一关系
- **Notification（通知记录）**: 告警触发通知后的历史记录，包含告警 ID、通知时间、通知内容等属性，与 Alert 是多对一关系
- **LogEntry（日志条目）**: 从 Loki 查询到的日志数据，包含时间戳、命名空间、Pod 名称、容器名称、日志内容等属性，用于规则匹配和告警样例展示

### Non-Functional Requirements

**性能**

- **NFR-001**: 规则引擎查询 Loki 的单次查询响应时间应在 5 秒内完成（正常网络条件下）
- **NFR-002**: Web 界面加载规则列表和告警列表的响应时间应在 2 秒内（数据量在 100 条以内）

**可靠性**

- **NFR-003**: 规则引擎必须支持故障恢复，重启后从数据库恢复告警状态和冷却期计时，确保告警状态的连续性
- **NFR-004**: 数据库操作失败时，规则引擎应记录错误日志并继续处理下一条规则，不应因单条规则失败而中断整体循环
- **NFR-005**: Loki 连接失败时，规则引擎应记录错误日志并在下一周期重试，不中断服务运行

**可维护性**

- **NFR-006**: 所有关键操作（规则匹配、告警生成、通知发送）必须记录结构化日志，包含时间戳、操作类型、相关实体 ID
- **NFR-007**: 配置参数（Loki URL、查询周期、数据库路径）应通过配置文件管理，支持修改后重启生效

## Clarifications

### Session 2026-02-04

- Q: 规则引擎查询周期 - 当前规格提到"规则引擎必须定期（默认每 30 秒）查询 Loki"，但在高频错误场景下，30 秒的延迟可能影响告警及时性。应该缩短至 10-15 秒提升实时性、保持 30 秒优先稳定性、还是支持可配置（10-60 秒）？ → A: 保持 30 秒，优先稳定性和资源消耗

- Q: 告警状态管理 - 当前规格中告警记录包含状态字段（active/resolved），但 MVP 阶段明确将"告警状态自动解决"列为不实现范围。告警状态如何变更？是所有告警保持 active、支持手动确认、还是基于时间自动解决？ → A: 所有告警保持 active，不实现 resolved 状态

- Q: 样例日志存储策略 - 当前规格要求告警详情页"展示至少一条样例命中日志"，但未明确如何存储这些样例日志。是仅存储最新一条、存储最近 N 条、还是不存储内容仅存储 Loki 查询链接？ → A: 仅存储最新一条命中日志的关键字段（时间戳、内容、pod 等）

- Q: 规则引擎故障恢复策略 - 当规则引擎因异常崩溃或重启时，未明确如何处理正在进行的告警状态和冷却期计时。是从数据库恢复状态继续计时、重启后重置所有冷却期、还是 MVP 阶段不处理？ → A: 从数据库恢复告警状态和冷却期计时，继续运行

- Q: Web 界面技术栈 - 当前规格要求"系统必须提供 Web 界面"，但未明确前端技术选型。是使用纯 HTML + 原生 JavaScript、React + UI 库、Vue.js + Element UI，还是其他方案？ → A: React + 轻量 UI 库（Ant Design 或 Material-UI）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员能够在 5 分钟内完成一条告警规则的创建（从打开 Web 界面到规则保存成功）
- **SC-002**: 当测试 Pod 输出匹配日志后，系统在 60 秒内生成告警记录并在 Web 界面可见
- **SC-003**: 窗口阈值规则能够正确合并同一分组维度的多次命中，告警记录的命中次数和最后触发时间准确更新
- **SC-004**: 冷却期机制能够有效控制通知频率，在冷却期内不产生重复通知，冷却期结束后能够再次通知
- **SC-005**: 系统能够稳定运行至少 1 小时，规则引擎持续查询 Loki 并处理日志，不出现崩溃或内存泄漏
- **SC-006**: Web 界面能够在 2 秒内加载规则列表和告警列表（数据量在 100 条以内）
- **SC-007**: 提供一个可重复执行的演示脚本，能够在 10 分钟内完成从部署测试 Pod、创建规则、触发告警到查看通知的完整流程验证

### Assumptions

- 本地 Mac 环境已安装并运行 Minikube，Loki、Promtail、Grafana 已部署且正常工作
- demo 命名空间已存在，Promtail 已配置采集该命名空间的容器日志
- Loki 的 HTTP API 可通过本地网络访问（如 http://localhost:3100）
- MVP 阶段为单用户使用，不考虑多用户权限和并发冲突
- 日志格式为容器标准输出（stdout/stderr），暂不支持结构化日志解析
- 规则引擎和 Web 服务运行在本地 Mac 环境，可以是独立进程或容器
- 数据库使用轻量级方案（如 SQLite 或嵌入式数据库），数据持久化在本地文件系统
- Web 前端使用 React + UI 组件库（Ant Design 或 Material-UI）构建，支持现代浏览器（Chrome/Edge/Firefox）
- 通知方式为控制台输出，后续阶段再扩展到邮件、钉钉、企业微信等渠道
- 规则引擎的查询周期默认为 30 秒，可通过配置文件调整
- 窗口阈值规则的时间窗口基于日志的时间戳，而非系统接收时间

### Out of Scope (MVP 阶段不实现)

- 静默（silence）功能：手动屏蔽特定告警的通知
- 序列规则或关联规则：基于多条规则的组合逻辑触发告警
- 复杂权限管理：多用户、角色、权限控制
- 分布式部署和高可用：规则引擎的集群部署、故障转移
- 通知渠道扩展：邮件、钉钉、企业微信、Webhook 等
- 通知重试和限流：通知失败重试、频率限制
- 规则测试 UI：在界面上模拟规则匹配和预览结果
- 结构化日志解析：JSON、键值对等格式的字段提取和匹配
- 告警状态自动解决：基于日志恢复或时间自动将告警标记为 resolved
- 告警统计和趋势分析：告警数量趋势图、Top 规则排行
- 规则模板和导入导出：预定义规则模板、批量导入导出规则配置
- 性能优化：大规模日志查询优化、缓存机制
