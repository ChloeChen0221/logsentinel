# 系统技术架构与组件选型

本项目采用 Kubernetes 云原生架构，结合日志采集、存储、规则引擎与告警通知，实现一个可扩展的日志告警平台。整体技术方案如下。

---

## 1. Kubernetes 环境

- **Minikube**  
  用于本地开发与调试，快速搭建轻量级 Kubernetes 集群。

- **（后期可选）老师服务器环境**  
  部署方式可选择：
  - `kubeadm`
  - `k3s`

  用于后续压测与最终展示，增强项目完整性与实用性。

---

## 2. 日志采集与存储

- **Promtail（DaemonSet 部署）**  
  在 Kubernetes 集群中以 DaemonSet 形式运行，采集每个节点的容器日志并转发到 Loki。

- **Loki（日志存储与查询）**  
  负责日志的集中存储与检索，支持：
  - LogQL 查询语言
  - HTTP API 调用

- **Grafana（日志可视化）**  
  提供日志展示与查询验证界面，便于演示、排错与用户使用。

---

## 3. 后端与服务划分（Python）

系统后端采用微服务式划分，基于 Python 构建核心逻辑。

- **FastAPI（API 服务）**  
  提供系统 RESTful API，包括：
  - 规则管理（CRUD）
  - 告警查询
  - 静默管理
  - 通知渠道配置
  - 简单鉴权与用户接口

- **Engine Worker（规则引擎服务）**  
  核心规则计算组件，负责：
  - 按规则游标从 Loki 拉取日志
  - 执行匹配、窗口统计、序列检测等逻辑
  - 去重与冷却控制
  - 生成告警事件

- **Notifier（通知服务，可选独立）**  
  可单独部署或集成在 Engine 中，实现：
  - 邮件/钉钉等通知发送
  - 重试机制与限流控制

---

## 4. 数据与状态存储

- **数据库（规则与配置存储）**
  - 开发阶段采用 **SQLite**（部署简单）
  - 后期可切换至：
    - PostgreSQL
    - MySQL  
    以提升并发与生产部署稳定性

- **状态存储：Redis**
  用于存储运行时状态信息，包括：
  - 日志游标
  - 窗口计数器
  - 序列检测状态
  - 去重与冷却状态

---

## 5. 前端 Web

- **React + Ant Design**
  前端采用 React 构建，结合 Ant Design 组件库，具备：
  - 生态成熟
  - UI 组件丰富
  - 快速搭建管理平台能力

---

## 6. 部署与交付

- **Helm 部署基础组件**
  Loki / Promtail / Grafana 等组件通过 Helm 安装与管理。

- **自研服务部署方式**
  - 初期使用 Kubernetes YAML：
    - Deployment
    - Service
  - 后期可进一步 Helm 化（加分项），增强交付能力与专业性

---
